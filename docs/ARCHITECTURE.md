# Архитектура приложения «Контроль пищевой безопасности»

## 1. Клиентская часть

Клиентская часть реализована как SPA на React.

Основные элементы:

- маршрутизация без перезагрузки страницы;
- страницы входа и регистрации;
- главная страница со списком проверок;
- карточка проверки;
- форма добавления и редактирования;
- личный кабинет;
- административный раздел пользователей;
- журнал действий;
- обработка ошибок 401 и 403;
- ограничение интерфейса по роли пользователя.

## 2. Серверная часть

Серверная часть реализована на Node.js + Express.

Она отвечает за:

- REST API;
- проверку логина и пароля;
- выдачу JWT;
- проверку токена на защищённых эндпоинтах;
- проверку ролей;
- работу с базой PostgreSQL;
- валидацию входных данных;
- журналирование действий;
- обработку ошибок.

## 3. База данных

Используется PostgreSQL. Базу можно открыть через pgAdmin. Таблицы:

- `users` — пользователи и роли;
- `inspections` — проверки пищевой продукции;
- `audit_logs` — журнал действий.

## 4. Средства безопасности

- bcrypt-хеширование паролей;
- JWT-аутентификация;
- разграничение доступа по ролям;
- защита API middleware-слоем;
- валидация входных данных;
- журналирование действий;
- обработка ошибок без раскрытия лишних технических деталей;
- рекомендации по HTTPS и backup.

## 5. Поток данных

1. Пользователь открывает React SPA.
2. SPA отправляет запрос на `/api/auth/login`.
3. Backend проверяет пароль и возвращает JWT.
4. SPA сохраняет токен и передаёт его в заголовке `Authorization`.
5. Backend проверяет токен и роль.
6. Backend выполняет операцию с базой данных.
7. Backend возвращает JSON-ответ.
8. SPA обновляет интерфейс без перезагрузки страницы.
