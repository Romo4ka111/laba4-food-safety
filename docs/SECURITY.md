# Меры безопасности

## Аутентификация

Пользователь входит по логину/email и паролю. После успешной проверки сервер выдаёт JWT-токен. Токен используется клиентом в заголовке `Authorization: Bearer <token>`.

## Авторизация и роли

В системе есть три роли:

- `user` — обычный пользователь;
- `specialist` — специалист ИБ;
- `admin` — администратор.

Административные эндпоинты `/api/users` и `/api/audit-log` доступны только роли `admin`.

## Хранение паролей

Пароли не хранятся в открытом виде. Используется `bcryptjs`, в базе сохраняется только `password_hash`.

## Хранение JWT

В учебной реализации токен хранится в `localStorage`, потому что это упрощает демонстрацию и интеграцию SPA. Риск: при XSS-атаке злоумышленник может прочитать токен. Для снижения риска в проекте не используется `dangerouslySetInnerHTML`, React экранирует выводимые значения, а данные валидируются на сервере.

Более безопасный production-вариант — хранить токен в `httpOnly Secure SameSite` cookie. Такая cookie недоступна JavaScript, но при использовании cookie нужно дополнительно учитывать CSRF-защиту.

## Защита от XSS

- не использовать вставку пользовательского HTML;
- валидировать и ограничивать длину строк на сервере;
- использовать React-экранирование;
- добавить Content Security Policy на уровне веб-сервера;
- не хранить лишние чувствительные данные в браузере.

## Защита от CSRF

При хранении JWT в заголовке `Authorization` риск CSRF ниже, так как браузер не отправляет этот заголовок автоматически. Если токен переносится в cookie, нужно включить `SameSite`, CSRF-токены и проверку Origin/Referer.

## Обработка ошибок

Клиент получает человекочитаемые сообщения. В production-режиме сервер не должен раскрывать стек ошибок.

## Журналирование

В журнал записываются:

- регистрация;
- успешные и неудачные входы;
- выход;
- создание, изменение и удаление проверок;
- изменение пользователей;
- попытки доступа без прав;
- запросы к защищённым API без токена;
- запросы с недействительным токеном;
- ошибки сервера.
