# Пользовательские сценарии

## Сценарий 1. Регистрация пользователя

1. Пользователь открывает `/register`.
2. Вводит имя, email и пароль.
3. Система проверяет корректность данных.
4. Backend проверяет уникальность имени и email.
5. Пароль хешируется.
6. Пользователь создаётся с ролью `user`.
7. Действие записывается в журнал.

Ошибки:

- слабый пароль — 400;
- занятый email — 409.

## Сценарий 2. Вход в систему

1. Пользователь открывает `/login`.
2. Вводит логин/email и пароль.
3. Backend проверяет хеш пароля.
4. При успехе сервер возвращает JWT.
5. SPA сохраняет токен и открывает главную страницу.

Ошибки:

- неверный пароль — 401;
- заблокированный пользователь — 403.

## Сценарий 3. Создание проверки

Право: `user`, `specialist`, `admin`.

1. Пользователь открывает `/add`.
2. Заполняет поля проверки.
3. Система валидирует данные.
4. Backend создаёт запись в таблице `inspections`.
5. Действие записывается в `audit_logs`.
6. Пользователь возвращается к списку.

## Сценарий 4. Редактирование проверки

Право:

- `user` — только свои проверки;
- `specialist` и `admin` — все проверки.

1. Пользователь открывает `/edit/:id`.
2. Backend проверяет JWT и права.
3. Пользователь меняет данные.
4. Backend сохраняет изменения.
5. Действие записывается в журнал.

## Сценарий 5. Удаление проверки

1. Пользователь нажимает «Удалить».
2. Интерфейс требует подтверждение.
3. Backend проверяет токен и права.
4. Запись удаляется.
5. Действие записывается в журнал.

## Сценарий 6. Управление пользователями

Право: только `admin`.

1. Администратор открывает `/admin/users`.
2. Меняет роль или статус пользователя.
3. Backend проверяет роль администратора.
4. Данные обновляются.
5. Действие записывается в журнал.

## Сценарий 7. Попытка доступа без прав

1. Обычный пользователь пытается открыть `/audit-log` или `/api/users`.
2. Frontend не показывает пункт меню.
3. Если запрос отправлен напрямую, backend возвращает 403.
4. Попытка записывается в журнал как `ACCESS_DENIED`.


## Сценарий: выгрузка проверок

1. Авторизованный пользователь открывает список проверок.
2. При необходимости задаёт поиск, фильтр и сортировку.
3. Нажимает кнопку «Выгрузить CSV».
4. Система формирует файл `inspections.csv` только по доступным пользователю данным.
5. Событие выгрузки фиксируется в журнале действий.
