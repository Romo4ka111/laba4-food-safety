# Техническое задание

## 1. Название проекта

Система учёта проверок безопасности пищевой продукции с разграничением доступа и журналом действий.

## 2. Предметная область

Приложение используется для учёта проверок предприятий пищевой промышленности. Система помогает фиксировать сведения о проверках, продукции, уровне риска, статусе и результатах контроля.

## 3. Проблема

Без централизованной системы данные о проверках могут храниться в разрозненных файлах, изменяться без контроля, теряться и быть доступными пользователям без необходимых прав. Это создаёт риски информационной безопасности: несанкционированное изменение записей, отсутствие журнала действий, слабый контроль доступа.

## 4. Цель разработки

Разработать защищённое веб-приложение для учёта проверок безопасности пищевой продукции с REST API, базой данных, JWT-аутентификацией, ролями пользователей и журналированием действий.

## 5. Пользователи и роли

| Роль | Возможности |
|---|---|
| user | регистрация, вход, создание проверок, просмотр и изменение своих записей |
| specialist | просмотр и обработка проверок, изменение статусов, работа с данными контроля |
| admin | управление пользователями, ролями, статусами и просмотр журнала действий |

## 6. Функциональные требования

1. Система должна поддерживать регистрацию пользователя.
2. Система должна выполнять вход по логину/email и паролю.
3. Система должна хранить пароли только в виде хеша.
4. Система должна выдавать JWT-токен после успешного входа.
5. Система должна защищать API от доступа без токена.
6. Система должна поддерживать роли user, specialist, admin.
7. Система должна позволять создавать, просматривать, изменять и удалять проверки.
8. Система должна поддерживать поиск, фильтрацию, сортировку, пагинацию и выгрузку CSV списка проверок.
9. Система должна фиксировать важные действия в журнале.
10. Система должна корректно обрабатывать ошибки и не раскрывать лишнюю техническую информацию пользователю.

## 7. Требования к безопасности

- JWT-аутентификация.
- Проверка роли на защищённых эндпоинтах.
- Хеширование паролей через bcryptjs.
- Валидация входных данных на клиенте и сервере.
- Единый формат ошибок.
- Журналирование входов, ошибок авторизации, изменений и удалений.
- Ограничение доступа к административным разделам.
- Рекомендация использовать HTTPS при деплое.

## 8. Интерфейс

Основные страницы:

1. `/login` — вход.
2. `/register` — регистрация.
3. `/` — главная страница со спинером, статистикой, фильтрами и списком проверок.
4. `/detail/:id` — карточка проверки.
5. `/add` — добавление проверки.
6. `/edit/:id` — редактирование проверки.
7. `/profile` — личный кабинет.
8. `/admin/users` — управление пользователями.
9. `/audit-log` — журнал действий.

## 9. Проверяемый результат

Результат считается достигнутым, если пользователь может зарегистрироваться, войти, получить JWT, выполнить CRUD-операции с проверками, а администратор может изменить роль пользователя и посмотреть журнал действий.
